**Speaker 2:** That is an incredible engineering failsafe. And I assume managing the lifecycle of these keys requires regular rotation to stay compliant right?

**Speaker 1:** Oh definitely. The docs mention you can set the key to auto rotate, where AVS detects a new version of the key within 10 minutes. Or you can lock the encryption to a specific key version URI if you want manual control over the rotation schedule. And because the DEK is doing the actual heavy lifting on the disks, rotating the master KEK doesn't require re-encrypting petabytes of data. No it just re-wraps the tiny DEK. Right, it's completely transparent to the virtual machines running on top of it.

**Speaker 2:** And from an auditing perspective, this entire infrastructure is built on a foundation of continuous validation. Microsoft governs AVS through the Security Development Lifecycle or SDL. **Speaker 1:** Okay so we have secured the data sitting dormant on the physical disks. If someone steals the drives they just have encrypted bricks. But what happens the millisecond a virtual machine actually powers on? That is the big question. The moment that VM boots, the hypervisor decrypts the disk and loads the operating system into memory. If the operating system itself is already infected with a bootkit, a piece of malware that loads before the antivirus software even starts, all of that disk encryption is completely useless. The malware is already inside the house. This is perhaps the most critical frontier in virtualization security. We have to move from protecting the storage to hardening the core compute execution. This brings us to a comprehensive security architecture known as Trusted Launch.

**Speaker 2:** Trusted Launch. Right.

**Speaker 1:** Trusted Launch is designed to protect the virtual machine from the ground up. And it relies on three foundational pillars. Secure Boot, the Virtual Trusted Platform Module or vTPM, and Virtualization Based Security, VBS. Let's start with Secure Boot. The concept has been around in physical laptops for years but applying it at scale in a massive cloud environment is fascinating. Secure Boot establishes what the documentation calls a root of trust. A root of trust means establishing an unbroken chain of verified software before handing control over to the operating system. When an AVS virtual machine powers on, the hypervisor loads a UEFI firmware environment, rather than the legacy BIOS. Secure Boot acts as a cryptographic gatekeeper within that firmware. How does it do that? It dictates that the bootloader, the OS kernel, and every single kernel level driver must be digitally signed by a trusted cryptographic authority like Microsoft or VMware.

**Speaker 2:** So if an advanced persistent threat manages to slip a rootkit onto the VM's hard drive, and that rootkit attempts to inject its malicious driver into the boot sequence, Secure Boot inspects the driver. It sees that the digital signature is either missing, forged, or just doesn't match the known good certificate. And Secure Boot simply halts the boot process. It refuses to load the unauthorized code. It stops the infection before the malware can even execute in memory.

**Speaker 1:** That's huge. But Secure Boot only verifies the signatures. It doesn't store the keys and it doesn't prove to the outside network that the boot was successful. That requires a dedicated cryptographic vault. That is the role of the second pillar, the Virtual Trusted Platform Module or vTPM.

**Speaker 2:** Now in a physical server, a TPM is an actual microchip soldered onto the motherboard. Right, an actual piece of silicon. It acts as a vault for cryptographic keys and a hardware random number generator. But if we are running thousands of virtual machines on a single physical host, we can't have thousands of physical chips. So the hypervisor has to emulate that hardware.

**Speaker 1:** Using VM encryption, the ESXi hypervisor provisions a software counterpart of a TPM 2.0 chip for every single virtual machine. But the architectural brilliance of the vTPM lies in its isolation. The vTPM acts as a secure vault for the VM's secrets, but it exists entirely outside the reach of the VM's guest operating system.

**Speaker 2:** I compare this to a diplomatic pouch. If a country establishes an embassy in a foreign nation, that embassy and its diplomatic pouches are sovereign territory. The pouch exists physically inside the host country, which in this analogy is our virtual machine's guest OS. The OS knows the pouch is there. But the local authorities, the operating system, have absolutely zero jurisdiction to open the pouch, inspect its contents, or extract the documents inside. The boundary is absolute. The operating system can send an instruction to the vTPM, asking it to perform a cryptographic operation, for example, asking it to decrypt a BitLocker drive key. The vTPM performs the math inside its own isolated enclave and hands the result back to the OS. But the OS can never, under any circumstances, extract the private keys from the vTPM. And the vTPM performs an even more critical function during the boot process. Attestation.

**Speaker 1:** Attestation sounds like a legal concept. How does a microchip testify? It creates a cryptographic chain of custody. As the VM boots, before any component is executed, its cryptographic hash is calculated. The UEFI firmware hashes the bootloader. The bootloader hashes the OS kernel. So it's measuring everything. Everything. These hashes are securely stored inside specific memory slots within the vTPM called Platform Configuration Registers or PCRs. So it's essentially taking a digital fingerprint of every piece of code as it loads, and logging it into an immutable ledger.

**Speaker 2:** Precisely. Once the VM is running, an external health monitoring system, like Microsoft Defender, can query the VM. The vTPM takes all of those PCR hashes, signs them with its own unforgeable endorsement key, and sends this attestation report to the monitor. And then the monitor checks the math. The monitor compares those fingerprints against known good values. It proves cryptographically that the VM has not been tampered with. If the hashes don't match, the network knows the VM is compromised, and Sentinel can automatically isolate it from the virtual network.

**Speaker 1:** And the documentation notes a massive operational benefit for AVS customers here. Because the vTPM itself contains secrets, the vTPM data file on the datastore must be encrypted. Usually this requires setting up an external Key Management Server. But in AVS, you don't have to configure any external key provider to use vTPM. Azure VMware Solution natively manages the key providers for you. It's completely integrated. For the user, it is basically a magical checkbox. You right click the VM, go to edit settings, click add new device, select Trusted Platform Module, and the hypervisor handles the complex cryptographic scaffolding behind the scenes. It democratizes military grade hardware virtualization security.

**Speaker 2:** But there is a very important operational caveat highlighted in the documentation regarding cloning VMs. Ah yes. The difference between vSphere 7 and vSphere 8. If you have a highly secure, vTPM enabled VM running as a template, and you want to deploy 10 web servers from it. If you are on VMware vSphere 7 and you clone that virtual machine, vSphere creates an exact, identical replica of the VM and its vTPM. It clones the cryptographic secrets. Which is a catastrophic security issue. If you want those clones to act as independent entities, you do not want 10 web servers running around your network with the exact same cryptographic identity sharing the same endorsement keys. They couldn't uniquely attest to their health.

**Speaker 1:** But in vSphere 8, VMware introduced advanced options to fix this. When you initiate the clone operation, the wizard allows you to choose to either copy the TPM for exact replication, which is useful for backups, or replace the TPM. Replacing it strips the old secrets and generates a brand new, empty vTPM for the cloned VM, allowing it to generate its own unique keys and establish its own independent root of trust.

**Speaker 2:** Which is why understanding the underlying hypervisor version is critical. And turning all of this on, Secure Boot and vTPM, unlocks the ability to use in-guest security features. You can finally enable BitLocker drive encryption inside the Windows virtual machine, just like you would on a physical corporate laptop, because BitLocker now has a secure TPM vault to store its volume master keys.

**Speaker 1:** And that brings us to the final and perhaps most complex pillar of Trusted Launch, VBS, or Virtualization Based Security. We talked about how the hypervisor sits below the operating system and isolates memory for different VMs. VBS takes that concept of hypervisor isolation and pushes it inside the guest operating system itself. To understand VBS, we have to look at how operating systems manage privilege. In a standard OS, the kernel has the highest level of privilege. It has absolute read/write access to all system memory. If a sophisticated attacker manages to exploit a vulnerability in the kernel and gains system level access, they effectively own the entire machine. They can scrape the memory for plain text passwords, they can disable the antivirus, they can do anything.

**Speaker 2:** I always thought the OS kernel operated at ring zero, the absolute highest privilege state in the CPU architecture. In a purely physical machine traditionally yes. But in a virtualized environment the hypervisor sits below the operating system. It operates at an even higher privilege level, what we sometimes conceptualize as ring negative one. VBS leverages this hypervisor authority to fundamentally change the architecture of the guest OS. How does it do that? Does it run a secondary OS? Essentially yes. VBS instructs the underlying ESXi hypervisor to carve out an isolated, highly secure enclave of memory within the virtual machine's RAM. It creates a specialized miniature operating system environment called the Secure World, which is completely restricted from the Normal World where the primary Windows kernel operates.

**Speaker 1:** So even if an attacker successfully drops a zero day exploit, compromises the Windows kernel, and gains complete ring zero system privileges in the normal world, they are still trapped. If they attempt to read the memory addresses assigned to the Secure World, the ESXi hypervisor intercepts that CPU instruction and blocks it. The malware literally cannot see or touch that memory space. The hardware enforced isolation is impenetrable.

**Speaker 2:** And Windows uses this Secure World to protect its most critical security processes. For example, it moves the Local Security Authority, the LSA, which handles user authentication and credential caching, into this isolated enclave. This specific implementation is known as Credential Guard. Which completely neutralizes one of the most devastating lateral movement techniques in cybersecurity, the pass the hash attack. Absolutely neutralizes it. If an attacker compromises a server, they usually run a tool like Mimikatz to scrape the RAM for NTLM hashes or Kerberos tickets left behind by administrators. But with VBS and Credential Guard active, those hashes aren't in the normal RAM. They are locked inside the Secure World. The attacker scrapes the memory and finds nothing.

**Speaker 1:** It also enables Hypervisor Protected Code Integrity or HVCI. VBS moves the process that verifies digital signatures into the Secure World. Even if a normal kernel is compromised, it cannot execute any new code or load any new drivers without asking the Secure World to verify the signature first. And because the Secure World is isolated, the malware cannot tamper with the verification process. It represents a fundamental shift from software based defense to hypervisor enforced hardware defense.
