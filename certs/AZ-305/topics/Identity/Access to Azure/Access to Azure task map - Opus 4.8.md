### Skill: Design authentication and authorization solutions

#### Task: Recommend a solution for authorizing access to Azure resources

Forum-discovery note: Public candidate discussions consistently emphasize requirement-driven choices among Azure RBAC built-in versus custom roles, the scope level for role assignments, PIM for just-in-time privileged access, managed identities for application access, and the Azure roles versus Microsoft Entra roles distinction. Reddit, Microsoft Q&A, and Tech Community posts were used only as discovery signals and every product was validated against official Microsoft documentation before inclusion.

Coverage notes:

- Azure RBAC is the primary documentation set for this task, but a complete recommendation spans the role model itself (roles and scope), least-privilege refinement (custom roles and ABAC conditions), privileged-access controls (PIM), workload authorization (managed identities), the enforcement layer (Conditional Access), and the scope hierarchy (management groups).
- The key distinction from the sibling "authorizing access to on-premises resources" task is the target: this task authorizes access to Azure control-plane and data-plane resources through Azure Resource Manager, so Azure RBAC, PIM for Azure resource roles, and managed identities dominate, rather than Application Proxy or hybrid access.
- Match the mechanism to the requirement: built-in roles first, custom roles only when built-ins don't fit, ABAC conditions to reduce assignment sprawl or add attribute-based checks, PIM for standing-privilege reduction, managed identities for credential-free workload access, and groups (not individual users) as the assignment target.
- Architecture guidance is more useful than any single product overview for this recommend-style task. The Well-Architected Framework covers workload-level least-privilege and critical-account design, while the Cloud Adoption Framework covers estate-wide, landing-zone-scale access management and separation of duties.
- Download first: Azure RBAC, Privileged Identity Management, Managed identities for Azure resources, and the Azure Well-Architected Framework security pillar. Add Azure management groups, Azure Policy, Conditional Access, and Azure Lighthouse when scope-hierarchy, guardrail, control-plane-protection, or cross-tenant requirements appear.
- Understand the boundary between authorization tools: Azure RBAC governs who can act on a resource, Azure Policy governs allowed resource state and can block actions (including identity-aware DenyAction rules), and deny assignments (created only by Azure, for example by deployment stacks) explicitly block actions even where a role grants them. These are complementary, not interchangeable.
- Lightly covered but testable adjacent topics include deny assignments and resource locks for protecting resources, Microsoft Entra ID Governance access reviews and entitlement management for periodic authorization recertification, ID Protection risk signals feeding Conditional Access, and the Global Administrator elevate-access path to the User Access Administrator role at the root scope.
