Coverage notes:

- Azure RBAC is the primary documentation set and should be downloaded first; its overview, built-in roles, scopes, best practices, custom roles, and conditions provide the core exam coverage.
- Azure Well-Architected Framework and Cloud Adoption Framework are more useful than a single product root for enterprise design tradeoffs such as segregation of duties, privileged access, landing-zone inheritance, and delegated administration.
- Follow the usual decision path of principal, role, and scope; prefer a built-in role at the narrowest practical scope, use groups for human access, managed identities for Azure workloads, PIM for privileged access, and custom roles only when built-in roles cannot meet least privilege.
- Distinguish management-plane permissions from data-plane permissions. Contributor can manage many resource settings without necessarily reading their data, while some management actions such as listing storage keys can indirectly enable data access.
- Treat Azure RBAC as additive. Narrower Reader assignments do not reduce broader Contributor access; ABAC conditions can constrain supported assignments, while deny assignments are generally system-created rather than a normal customer-designed exclusion mechanism.
- Service-specific authorization remains important. Storage can involve Entra-based RBAC, SAS, keys, and Data Lake ACLs, while Key Vault supports the recommended RBAC model and a legacy access-policy model.
- Azure Lighthouse is the main cross-tenant resource-management option. Access reviews, Data Lake ACLs, Azure Policy guardrails, resource locks, and Defender for Cloud recommendations are potentially testable adjacent areas but are covered more lightly here.
- Do not confuse Azure RBAC with Microsoft Entra roles: Azure roles authorize Azure resource access, whereas Microsoft Entra roles administer directory resources. Classic subscription administrator roles are retired and should not drive a new design.
