Forum-discovery note: Public candidate discussions commonly emphasize requirement-driven choices among built-in and custom Azure roles, assignment scope, PIM for just-in-time privilege, managed identities for workload access, and the distinction between Azure roles and Microsoft Entra roles. Public discussions were used only as discovery signals; included topics were validated against official Microsoft documentation.

Coverage notes:

- Azure RBAC is the primary documentation set, but complete coverage spans privileged access, workload identities, architecture guidance, and service-specific data-plane authorization.
- Download Azure RBAC first, followed by Microsoft Entra ID Governance, Managed identities for Azure resources, and the Azure Well-Architected Framework; add Cloud Adoption Framework for enterprise landing-zone scenarios.
- Match the mechanism to the requirement: prefer a built-in role at the narrowest practical scope, assign human access through groups, use managed identities for workloads, use PIM for privileged access, add supported ABAC conditions for finer control, and create custom roles only when built-in roles cannot meet least privilege.
- Distinguish Azure resource roles from Microsoft Entra directory roles, and distinguish management-plane Actions from data-plane DataActions. Contributor access to resource configuration does not automatically imply data access, although some management actions can expose credentials or keys.
- Azure RBAC is additive: a narrower Reader assignment does not reduce broader Contributor access. ABAC conditions can constrain supported role assignments, while deny assignments are generally created by Azure-managed features rather than used as a normal customer-designed exclusion mechanism.
- Service-specific authorization remains important. Storage can involve Entra-based RBAC, SAS, Shared Key, and Data Lake ACLs, while Key Vault supports the recommended RBAC model and a legacy access-policy model.
- Azure Lighthouse is the principal cross-tenant resource-management option. Conditional Access, management groups, Azure Policy, resource locks, access reviews, ID Protection, and Defender for Cloud are useful adjacent controls but should not displace the core authorization sources.
- This task targets Azure control-plane and data-plane resources; Application Proxy and other hybrid access mechanisms belong primarily to the sibling task for authorizing access to on-premises resources.
