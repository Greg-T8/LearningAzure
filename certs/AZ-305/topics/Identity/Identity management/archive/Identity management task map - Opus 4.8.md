Forum-discovery note: Public candidate discussions consistently identify Microsoft Entra ID, hybrid identity, external identities, tenant design, and identity lifecycle as important AZ-305 coverage, and frequently also mention RBAC, PIM, and access reviews. Those signals were used only for discovery and validated against official Microsoft documentation; RBAC, PIM, and access reviews were treated as adjacent authorization and identity-governance objectives rather than the center of this task.

Coverage notes:

- Microsoft Entra ID is the primary product family, but the recommendation is fragmented across core directory, hybrid identity, External ID, provisioning, multitenant architecture, device identity, and Domain Services documentation sets — no single product root covers the whole task.
- Start with Azure Architecture Center and Azure security for the design lens, then download Microsoft Entra ID, Hybrid identity, Microsoft Entra External ID, Application provisioning, and Microsoft Entra Domain Services for the principal service-selection decisions.
- Prefer one workforce tenant and one authoritative identity source when requirements allow. Choose cloud-only identities for cloud-first organizations; integrate Active Directory through synchronization when users must retain a common identity across on-premises and cloud resources.
- Microsoft Entra Cloud Sync is Microsoft's strategic direction for most hybrid synchronization scenarios, but use the decision guide before replacing Connect Sync, because some advanced or large-directory requirements can still require Connect Sync.
- Use B2B collaboration when partners should use identities managed by their home organizations, and an external tenant for consumer or business-customer CIAM. Treat Azure AD B2C as a legacy consideration only: it has not been available to new customers since May 1, 2025, though Microsoft states support continues until at least May 2030.
- Use automated app provisioning for identity lifecycle in SaaS or private applications, and HR-driven provisioning when an HR platform should be the source of authority.
- Use multiple workforce tenants only for requirements such as mergers, sovereignty, administrative isolation, or strong resource and identity boundaries that delegation within one tenant cannot satisfy; account for cross-tenant provisioning and operational complexity.
- Use Microsoft Entra Domain Services for legacy domain protocols without operating domain controllers, and retain self-managed AD DS when a workload needs capabilities outside the managed service. Device identities are a lighter but testable part of the overall identity model.
- Authentication methods, Conditional Access, Identity Protection, Azure RBAC, Microsoft Entra roles, PIM, access reviews, entitlement management, managed identities, and Workload ID remain important adjacent topics, but the current AZ-305 guide assigns authentication, authorization, application identities, and identity governance to separate tasks.
