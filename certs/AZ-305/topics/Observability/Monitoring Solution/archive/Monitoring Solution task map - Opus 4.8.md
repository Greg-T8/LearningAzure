Coverage notes:

- Coverage is concentrated in Azure Monitor but fragmented across its feature areas, because "monitoring" spans the data platform (metrics versus logs), the insights layer (VM, Container, Network), application APM, visualization, alerting, and automated response. Downloading one Azure Monitor PDF covers most rows here.
- The distinguishing decision for this task, versus the sibling logging and routing tasks, is what you do with telemetry after collection: alert, visualize, autoscale, or escalate. Expect scenarios that state a symptom and requirement and ask which service or alert type satisfies it.
- Alert-type selection is the highest-yield subtopic: metric alerts for near-real-time numeric thresholds with dynamic thresholds, log search alerts for KQL across any workspace data, activity log alerts for Service Health and Resource Health, and Prometheus alerts for PromQL on Kubernetes.
- Platform-versus-resource health is a common gap: Service Health and Resource Health answer "is Azure broken" while Azure Monitor answers "is my workload broken." A recommendation that only covers resource telemetry misses planned maintenance and health advisories.
- Download first: Azure Monitor, Azure Well-Architected Framework, Azure Service Health, Microsoft Defender for Cloud, and Network Watcher. Add Monitoring and health when the scenario is identity-centric, and Azure Managed Grafana when it is Kubernetes or cloud-native.
- Architecture guidance outweighs any single product root, because a monitoring recommendation combines signal type, scope, audience, response action, retention, and cost into one answer. The Well-Architected monitoring design guide and the OE:07 observability guidance are the strongest framing for that.
- Lightly covered but testable adjacent topics include Azure Advisor recommendations across reliability, security, performance, cost, and operational excellence; Azure Policy and Azure Monitor Baseline Alerts for deploying alert rules at scale across a landing zone; Microsoft Sentinel when monitoring extends into SIEM correlation; and Cloud Adoption Framework guidance for monitoring an entire cloud estate rather than a single workload.
- Azure Monitor health models are in preview and introduce state-based, health-rollup alerting as an alternative to resource-centric alert rules. The Well-Architected guidance now leans on them heavily, so they are worth knowing conceptually even though preview features appear on the exam only when commonly used.
