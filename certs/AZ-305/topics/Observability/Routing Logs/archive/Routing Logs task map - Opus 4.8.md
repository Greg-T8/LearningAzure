Coverage notes:

- This task is anchored in Azure Monitor, but the useful documentation is fragmented across the routing mechanisms (diagnostic settings, DCRs, agents, the pipeline, data export, and Policy enforcement) and the destination services (Log Analytics, Storage, Event Hubs, Sentinel, partner solutions).
- The central design decision is matching the routing mechanism to the log source: diagnostic settings for platform, activity, resource, and Microsoft Entra logs; DCRs plus Azure Monitor Agent for guest OS and workload logs; the Azure Monitor pipeline for edge, on-premises, and multicloud ingestion; and Log Analytics data export for post-ingestion routing.
- Destination selection is the other half of the decision: Log Analytics for query, alerting, workbooks, and Sentinel; Storage (ideally immutable) for cheap, long-term, compliance-grade archive; Event Hubs for streaming to external SIEM or analytics; Sentinel for security operations; partner solutions for third-party observability platforms.
- Note the routing constraints that appear in scenarios: a maximum of five diagnostic settings per resource, one destination of each type per setting, same-region requirements for Storage and Event Hubs destinations, and trusted-Microsoft-services firewall bypass when networks are locked down.
- Download first: Azure Monitor (diagnostic settings, resource logs, activity log, DCRs, data export), Monitoring and health, Azure Event Hubs, Azure Blob Storage, Microsoft Sentinel, and Network Watcher.
- Architecture guidance outweighs any single product root here, because routing questions combine source type, destination, retention, region, access model, cost, and compliance into one recommendation.
- Lightly covered but testable adjacent topics include transformations for cost control at ingestion, Azure Policy remediation for enforcing routing on existing resources, Azure Arc for hybrid routing, Azure Monitor partner solutions for ISV destinations, and Microsoft Defender for Cloud as both a producer and consumer of routed security data.
