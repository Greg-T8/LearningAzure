### Comparison: Device Identity States

| Feature | Microsoft Entra Registered | Microsoft Entra Joined | Microsoft Entra Hybrid Joined |
| :--- | :--- | :--- | :--- |
| **Primary Target** | Bring Your Own Device (BYOD) / Personal | Corporate-owned devices in a cloud-first model | Corporate-owned devices with legacy on-premises AD dependencies |
| **Trust Scope** | Exists only within the specific user's profile | Exists with the entire device | Exists with the entire device |
| **Sign-in Account** | Local credential, personal Microsoft account, local PIN | Organizational Microsoft Entra account | Organizational Active Directory account |
| **Management** | Unmanaged (unless explicitly enrolled in MDM) | Cloud-based management (e.g., Intune) | Co-managed (Intune + Group Policy/Configuration Manager) |
| **Unsupported Features** | Unified Write Filter | Requires Entra Connect for on-premises SSO | Device/user password changes and TPM resets without on-premises line-of-sight |

### 1. Core Device Identity States

**Microsoft Entra Registration**

* **Authentication Flow:** Users sign in with local credentials or standard Windows Hello (local PIN/biometrics) and add their Microsoft Entra account as a secondary account for SSO.
* **Exam Trap:** Do not confuse standard Windows Hello (supported on Registered) with Windows Hello for Business (explicitly designed as a primary sign-in for Joined/Hybrid devices).
* **Limitation Checklist:**
  * Cannot use the **Unified Write Filter** feature.
  * Organization cannot inherently control the device unless it undergoes separate Mobile Device Management (MDM) enrollment.

**Microsoft Entra Join**

* **Scenario Example:** Your organization is purchasing new Windows 11 devices and wants to adopt a cloud-first management model using Microsoft Intune. Microsoft Entra Join is the primary and recommended choice.
* **Exam Trap:** A common misconception is that "hybrid join" is required to access on-premises file shares. Microsoft Entra joined devices **do** support SSO to on-premises resources.
* **On-Premises Access Configuration:**
  * The cloud-joined device has no innate knowledge of the local AD.
  * You must use **Microsoft Entra Connect** to synchronize default attributes (Domain Name, SAM Account Name, UPN) to the cloud.
  * Entra ID passes this data down with the user's Primary Refresh Token (PRT).
  * The Local Security Authority (LSA) uses this to locate a domain controller and acquire a Kerberos TGT or NTLM token.

**Microsoft Entra Hybrid Join**

* **Dependencies:** Requires a constant network connection/line-of-sight to on-premises domain controllers.
* **Exam Trap:** Certain actions will completely fail if the user is remote and off the corporate network (without VPN). This explicitly includes **device password changes, user password changes, and Trusted Platform Module (TPM) resets**.
* **Scenario Example:** A remote user changes their password via Entra self-service password reset (SSPR). The physical hybrid device cannot validate this without domain controller access; the user must sign in using their old cached password.

### 2. The Registration and Authentication Architecture

**Components and Services**

* **Dsreg (Registration):** The Windows component that handles the device registration process across all device states.
  * Generates cryptographic key pairs and binds private keys to the hardware TPM.
* **Local Security Authority (LSA) (Authentication):** Handles Kerberos and NTLM authentication by passing user credentials from the sign-in screen to the CloudAP plugin.
  * **Exam Trap:** Do not assign device registration tasks to LSA. LSA is for authentication; Dsreg is for registration.
* **Azure Device Registration Service (DRS):** The cloud engine that finalizes registration.
  * It validates the user's ID token, creates a unique device ID, and issues a device certificate based on the client's request.

**Cryptographic Keys and Certificates**

* **Device Key (`dkpub` / `dkpriv`):** Generated by Dsreg (ideally bound to the TPM) and used specifically to **create and sign a certificate request** sent to Azure DRS.
* **Transport Key (`tkpub` / `tkpriv`):** Derived from the TPM's storage root key immediately after the device key is created, used to secure ongoing token exchanges.
* **`MS-Organization-Access` Certificate:**
  * The foundational cryptographic trust issued by Azure DRS.
  * **Configuration Detail:** Stored in `Local Computer\Personal\Certificates` (Joined/Hybrid) or `Current User\Personal\Certificates` (Registered).
  * **Exam Trap:** Do not confuse its lifetime with a PRT. The `MS-Organization-Access` certificate has a **10-year default lifetime**, while a PRT is 90 days. If the certificate is deleted, the device faces authentication failures and must be re-registered.

**Primary Refresh Tokens (PRTs)**

* PRTs provide Single Sign-On (SSO) and are valid for **90 days**.
* **Registered Device PRTs:** Bound to an Entra identity and a hardware TPM. **They satisfy device registration policies for Conditional Access**.
* **Unregistered Device PRTs:** Issued when signing in via a browser on a personal device without an Entra identity. Not bound to a TPM. **Cannot satisfy Conditional Access policies**.
* **Troubleshooting Detail:** Run `dsregcmd /status` on a Windows device and check the **`AzureAdPrt`** field under the "SSO State" section. If it says **NO**, check the "Attempt Status" field directly below it for the specific error code (e.g., `0xc000006d`).

### 3. Microsoft Entra Hybrid Join Mechanics

**Service Connection Point (SCP) Configuration**

* Devices query on-premises AD for the SCP to find tenant routing information.
* **Configuration Detail:** During manual setup or GPO deployment, the SCP must contain two specific keywords:
  * **`azureADid`**: The Microsoft Entra Tenant ID.
  * **`azureADName`**: The verified domain name (dictates federated vs. managed routing).
  * **Exam Trap:** Incorrectly choosing `enterpriseID` or `registrationURL` instead of the required `azureADid` and `azureADName` keywords.

**The `userCertificate` Attribute Trigger**

* During the Automatic Device Join task (managed environments), the local Windows device generates a self-signed certificate.
* **Configuration Detail:** It writes this certificate to the **`userCertificate`** LDAP attribute on its AD computer object.
* **Exam Trap:** Do not select `cloudAnchor`. `userCertificate` is the correct LDAP attribute that triggers Entra Connect to sync the device to the cloud.

**Managing the "Pending" State**

* **Normal Behavior:** Entra Connect syncs the device object to the cloud (putting it in Pending state) before the physical device reaches out to negotiate keys with Azure DRS. Pending devices cannot get PRTs or satisfy Conditional Access.
* **Troubleshooting Detail:** If an admin moves a device's OU out of the sync scope, Entra deletes it. If moved back, Entra Connect recreates it as "Pending", but the physical device thinks it is already registered.
  * **Resolution:** Run `dsregcmd /leave` on the local device and restart to force a fresh registration.

**Disabling Hybrid Devices**

* **Exam Trap:** Disabling a hybrid device only in the Entra admin center will cause Entra Connect to automatically re-enable it during the next sync cycle.
* **Resolution:** You must disable the computer account in the **on-premises Active Directory** first.

### 4. Networking and Federation Requirements

**Federated Environments (AD FS)**

* **Protocol Differences:** `WS-Fed` is used to *join* the device, but the **`WS-Trust`** protocol is specifically required for users to *sign in* and acquire a PRT.
* **Configuration Detail:** Administrators must enable specific WS-Trust endpoints: `usernamemixed` (username/password) or `certificatemixed` (smartcard/certificate).

**Proxy and TLS Limitations**

* **Scenario Example:** An organization uses a proxy server for "TLS break-and-inspect" data loss prevention.
* **Exam Trap:** This breaks the direct **client certificate authentication** required by Azure DRS.
* **Resolution:** Administrators must bypass registration URLs (e.g., `https://enterpriseregistration.windows.net` and `https://device.login.microsoftonline.com`) from TLS inspection.

### 5. Managing Devices and Policies

**MDM Enrollment Logic**

* Configured under "Mobility (MDM and MAM)" in the Entra admin center.
* **Scenario Example 1:** User is **IN** the MDM scope. If MDM enrollment fails during an Entra join, **the entire Microsoft Entra join is rolled back** to prevent non-compliance.
* **Scenario Example 2:** User is **NOT IN** the MDM scope. The join completes successfully, but the device is marked as **Unmanaged**.
* **Prerequisite:** Scoped users require an appropriate MDM license (e.g., Intune).

**Local Administrator Rights Limitations**

* Default security principals added during a join: 1) Entra Joined Device Local Administrator role, 2) Global Administrator role, 3) The user performing the join.
* **Exam Trap:** The "Reports Reader" role is NOT granted local admin rights.
* **Exam Trap:** Entra ID does not use a "primary group RID" to determine rights. Rights are passed via the PRT.
* **Limitation Checklist:** Windows sign-in supports the evaluation of a maximum of **20 Microsoft Entra groups** (including nested groups) for local administrator rights. Exceeding this causes privilege assignment failures.

**Device Stale States and Timestamp Evaluation**

* `ApproximateLastSignInDateTime` updates on the overview page every few hours, but the underlying data has a **14-day (+/- 5 day) update variance**.
* **Troubleshooting Detail:** Do not consider a timestamp younger than **21 days** as a reliable indicator of a stale device. Use the **sign-in audit logs** for real-time activity metrics.

**Deleted Device Recovery**

* Deleting a device in the admin center is **nonrecoverable**.
* The physical device gets flagged with `AadRecoveryEnabled=YES` and users are blocked from cloud resources.
* **Resolution:** Requires manual re-registration.
  * **Method 1:** Built-in Windows recovery flow triggers on next sign-in, forcing re-authentication.
  * **Method 2:** Sign in with local offline admin, manually disconnect the account under `Settings > Accounts > Access Work or School`, and rejoin.

**Enterprise State Roaming (ESR)**

* **Licensing Note:** Requires Microsoft Entra ID P1 or P2, or an Enterprise Mobility + Security (EMS) license.
* **Configuration Detail:** Requires Windows 11 or Windows 10 (21H2+). Encrypts data in transit and at rest.
* **Exam Trap / Architecture Shift:** As of May 2026, ESR management is migrating away from the Entra portal and will be strictly managed via **Windows Backup policies** (Intune) by the end of June 2026. Requires additional Microsoft 365 / Windows E3/E5 licenses.

### 6. Non-Windows Operating Systems

**macOS Platform SSO**

* **Prerequisites Checklist:**
  * **Intune Company Portal:** Must be installed to act as the authentication "broker" (acquires the PRT).
  * **MDM Enrollment:** Required because Apple's architecture only allows the SSO Extension profile to be pushed via an MDM payload.
* **Troubleshooting Detail:** If a user misses the "Registration Required" prompt, the built-in timer waits **10 minutes** before re-prompting.
  * **Manual Fix:** Users can sign out of the Mac and sign back in to force the prompt. For macOS 14+, go to Settings > Users & Groups > Network Account Server > Edit > Repair.

**Linux Devices (Microsoft Identity Broker)**

* **Upgrade Scenario:** Upgrading the Linux broker from version 1.6 to 3.0.1 crosses an architectural threshold (Java-based to C++-based).
* **Troubleshooting Detail:** An in-place upgrade will cause authentication failures. Administrators must perform a clean reinstall.
* **Exam Trap:** Because previous states are wiped, users **must re-register and re-enroll their devices**.
* **Configuration Detail:** The new 3.0.1 broker performs a **Microsoft Entra join** on the Linux device, creating device-wide trust instead of just profile-level registration.
